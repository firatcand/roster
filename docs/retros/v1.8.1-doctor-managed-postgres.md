# Retro — v1.8.1: doctor on managed Postgres (ROS-161)

**Shipped:** 2026-07-09 (PR [#314](https://github.com/firatcand/roster/pull/314), single-patch release)
**Scope:** `no-inbound-members` doctor check whitelists a superuser or full-brain-owner inbound member; `templates/env.example` documents the brain + brain-fs AWS vars.

## What happened

Deploying v1.8.0 to the owner's live Neon brains, `brain doctor` read UNHEALTHY on a single check — `no-inbound-members` — while everything else (including the new `s3-file-drift`) was green. Investigation on the live brain proved it unfixable at the SQL layer: Neon's control plane force-grants the DB owner (`neondb_owner`) into every role via its internal `cloud_admin` superuser, and PG16 lets only `cloud_admin` revoke that grant — `neondb_owner` can't even `SET ROLE cloud_admin`. ROS-154 had already found this and chosen to "report honestly + print remedial superuser SQL," but that SQL is unrunnable on a managed host. So the right fix was the check itself: an inbound member that already has ≥ the runtime role's privileges is not an escalation and shouldn't be flagged.

## What went well

- **RCA before the fix.** Rather than blindly relaxing the check, we reproduced the exact `pg_auth_members` state on the live brain (grantor `cloud_admin`, `admin_option`), tried the revoke, and watched it fail with the precise permission error. That turned "doctor is annoying" into a proven, documented root cause.
- **The review drove the design to soundness.** Codex's impl review (7→7→8) refused to accept the first two whitelist attempts: schema-ownership (schema ≠ objects) and then table-ownership (misses schema `USAGE` + `EXECUTE` on the SECURITY DEFINER brokers). The final whitelist requires ownership of the *entire* runtime-touched surface — both schemas, every table/view, every function — which maps 1:1 to what membership would confer. A weaker check would have shipped a real (if narrow) escalation blind spot.
- **Non-circular signal.** `has_table_privilege` would have been circular — the member inherits the runtime grants *through* the very membership under test. Ownership (`nspowner`/`relowner`/`proowner`) is independent of it. Naming that explicitly in the code comment is the kind of thing that saves the next reader.
- **Verified against the real brain, both ways.** The fixed doctor flipped both live Neon brains to `brain healthy`, and the tests pin the contract (full-owner → green; superuser → green; schema-only / table-gap / plain → flagged).

## What was hard

- **Getting the test to encode the *safe* case, not the hole.** The first "schema-owner is whitelisted" test literally proved the escalation Codex was worried about. The fix was to make the positive test transfer the *whole* brain's ownership (a scoped `ALTER` sweep — a blanket `REASSIGN OWNED` trips on postgres-pinned system objects) and to add a partial-owner test that stays flagged.

## What we'd do differently

- **A doctor check that encodes a security invariant should whitelist by "already ≥ the target's privileges," not by a convenient proxy (schema/DB ownership).** ROS-154's strict stance was defensible for self-hosted Postgres; the managed-PG reality needed the privilege-equivalence framing from the start.
