## Decisions that stuck

- **Full REVOKE, one-way door accepted** (owner call): the runtime role sits above the DB owner on Neon; no current flow needs SQL-level ALTER/DROP over it. Init **verifies** the revoke instead of trusting it and reports `creatorGrantRemains` honestly — doctor never lies to stay green.
- **Age since superseded, not since written** (owner call, deviating from the ticket's tentative default): a 3-year-old fact replaced yesterday stays recoverable for the full retention window. Later strengthened to *both-older* (see below) — strictly more conservative.
- **Edges dropped from gc** (owner call after Codex plan review): `brain get` renders raw edge rows as the entity timeline — superseded edge versions are user-visible history, not hidden clutter. Pruning them would need a read-contract change first.
- **Retention is policy → durable**: `--older-than` flag > `gc.retention` config key > 730d built-in.

## What surprised us

- **Stock PG16 records the CREATEROLE auto-grant as bootstrap-granted** — the creator's `REVOKE … FROM CURRENT_USER` is a silent warning-level no-op and `GRANTED BY` is permission-denied. Neon (PG 17.10) *can* revoke. Codex caught this from docs at plan-review score 3/10; one local psql session with a non-superuser CREATEROLE admin confirmed it. **Protocol keeper: when a reviewer cites Postgres semantics against your design, reproduce empirically before arguing.**
- **"Delete only old superseded versions" hides a fixpoint problem.** Four successively narrower counterexamples (batch recomputation, rerun re-anchoring, divergent mount/chunk timestamps, divergent denormalized `source_path`) all reduced to the same shape: deletion changes the adjacency you computed eligibility from. The stable predicate is *both the row and its immediate replacement older than retention*, anchored on the entity that controls re-anchoring (the mount group for chunks) — then a deleted row was necessarily old, a recent anchor can never vanish, and reruns can only shrink the set. Every counterexample is now a regression test with rerun assertions.
- **The 4-round Codex cap can land at 7/10 with all findings fixed** — the cap ended on a scoring technicality, not open issues. Owner-approved a 5th round: 9/10, zero findings. Worth keeping the escalation gate; also worth remembering convergence trajectory (5→6→7→7) is signal, not noise.
- `pg` returns `bigint` as **string** — `Number()`-coerce `RETURNING id` in test helpers before array comparisons, or `includes()` fails silently-green/red in confusing ways.

## What we'd do differently

- Write the *stability-under-deletion* argument into the plan before implementing any GC/pruning eligibility query — three of five review rounds were variations of that one omission.
- Stage test fixtures with divergent/non-monotonic metadata from the start when correctness depends on ordering assumptions; monotonic fixtures made early rounds look safer than they were.
