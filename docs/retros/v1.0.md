v1.0 drops the `projects/<slug>/` multi-tenant layout in favour of one workspace per repo. The same release introduces shared brand/voice substrate under `config/` + `guidelines/`, adds agent-level `.env` inheritance with explicit-unset semantics, collapses `chief-of-staff` from 11 plans to 4, removes the `promotion-arbiter` subagent from `dreamer` (single-playbook model), drops the `project` field from `roster/<function>/schedules.yaml`, and rewrites the public docs (HOWTO, ARCHITECTURE, README, API) for the v1 shape. Three new doctor checks (13, 14, 15) audit the merged agent-env shape. Breaking — there is no automatic migration; existing v0.4 workspaces must re-scaffold and copy state files (see [CHANGELOG § Migration](../../CHANGELOG.md#migration)).

## Decisions made (and why)

1. **Clean break, no migration tool.** v0.4 → v1.0 is a semver major and the single known v0.4 workspace was the maintainer's. `roster init` hard-rejects a `projects/` directory (ROS-81). No auto-strip, no warning-only mode. Cheaper to enforce one shape than to support both behind a flag.
2. **One opinion, one workspace.** `projects/<slug>/` was YAGNI inherited from `agent-team`'s earliest scaffold; the v1 cut collapses it everywhere. The single decision ripples through: 11 chief-of-staff plans → 4 (ROS-77, 90), `dreamer` loses the `promotion-arbiter` subagent (ROS-91), the schedule schema drops the `project` field (ROS-80), every slash command drops the trailing `for <project>` (ROS-89, 92), and the SDR worked example exits the shipped surface (pre-v1, #120). One opinion is a force-multiplier when the upstream shape changes.
3. **`config/` and `guidelines/` are split by edit cadence, not by content type.** `config/project.yaml` is machine-readable identity (name, stage, audience, motion) — edited rarely, parsed by the CLI. `guidelines/*.md` is human-readable prose (voice, messaging, brand-book, asset-links, ICP personas) — edited often, read by LLMs. Two files, two roles, no overlap. The split survives the next time we want to add a machine field or a new prose surface.
4. **Agent-level `.env` inheritance with empty-string = explicit unset.** `<agent>/.env` inherits from the workspace `.env`; an agent value overrides; an *empty* string in the agent file removes an inherited key (distinct from absent, which keeps the inherited value). Two visually-similar states with distinct semantics, both audited by doctor checks 14 + 15.
5. **Doctor checks must follow the shape they audit.** Checks 13 / 14 / 15 (audit the merged agent-env shape) had to land *after* the env-merge loader (ROS-84). When check 12 / 15 shipped against a walker that still resembled v0.4, they silently passed; ROS-99 and ROS-101 fixed the walker only after a real v1 fixture exercised them. Audit code is shape-aware; its fixtures must be too.
6. **Encode contract decisions in `conventions.md` before the skills are rewritten.** ROS-74 (Phase v1-1) rewrote `templates/scaffold/conventions.md` end-to-end for v1. The skill rewrites in Phase v1-3 (`chief-of-staff` ROS-90, `dreamer` ROS-91, `roster-orchestrator` ROS-89, `lesson-drafter` ROS-92) then implemented against the new contract instead of rediscovering it. Sequence the contract first.
7. **`v1.0-refactor` is a Linear label, not a project.** All 27 tickets stayed in the existing `roster` Linear project under one shared label. Cycle and milestone reporting work; the label survives v1.0 ship as an audit trail. Spinning a separate project per release would have doubled context-switch overhead with no benefit.

## Effort vs original plan

`spec/SPEC.md` carried a 1-week ballpark for the v1 refactor pre-decomposition. After `/decompose` cut the 4 phases into 27 tickets, actual wall-clock from first v1 ticket merge (ROS-72 / #150 at 2026-05-21 15:18Z) to last v1 code merge (ROS-90 / #178 at 2026-05-22 15:11Z) was ~24 hours across two parallel `claude` sessions. Average per ticket: ~1 hour of chained `/pickup-task` → `/plan-task` → `/implement` → `/ship` → review on the happy path. That is the upper bound of speed for a well-decomposed refactor at this codebase size — most tickets were 50-200 LOC diffs reshaping pre-existing patterns rather than greenfield work. Compression vs the human-team estimate sits at the ~30x feature-implementation row in `~/.claude/CLAUDE.md`'s table.

## Learnings to harvest

1. **A single contract document ahead of implementation collapses a multi-phase refactor.** Phase v1-1's first hour wrote `templates/scaffold/conventions.md` (ROS-74) — every later phase executed against that contract instead of rediscovering it. The 4-phase plan worked because the contract was settled; the work was reshaping code to fit it. For the next breaking-shape release: settle the contract before opening the first implementation ticket, not while landing it.
2. **Doctor / audit code must follow the shape it audits, not lead it.** ROS-99 and ROS-101 fixed checks that had silently passed against the v0.4 walker. Generalize: when a check is shape-aware, its test fixture must exercise the new shape *before* the check ships, not after. A passing check against the old fixture is not evidence the check works.
3. **Parallel sessions amplify decomposition quality.** Two `claude` sessions on a well-decomposed phase reveal whether the dependency graph is real or aspirational. The ROS-76 → ROS-74/79 cleanup gap surfaced precisely because two sessions hit the same test file from different tickets — the inter-ticket coupling was hidden by serial execution. Serial execution doesn't catch latent coupling; parallel execution does.
4. **"Delete X" tickets must own the test cleanup for X.** ROS-76 deleted plans + scripts; the tests that exercised them stayed in until ROS-74 and ROS-79 bundled the fix mid-flight. The lesson generalizes: every deletion ticket's acceptance must include "and update or delete tests that reference X". Cheap to add to the ticket; expensive to discover after merge.
5. **Codex `/second-opinion` is high-leverage on assertion design.** Already in `phase-v1-1.md`; restated here for the release-level audience. Any PR whose primary diff is assertions deserves an outside reviewer — the author's intuition about "this passes therefore it's right" is the failure mode the reviewer catches.
6. **CHANGELOG-as-contract is cheaper than per-phase retros.** v1.0 shipped 27 tickets with one CHANGELOG section (ROS-95 / #177), one phase-v1-1 retro covering the first phase's rationale, and this release retro rolling up the rest. Three documents instead of five. Optimization only works when the CHANGELOG is rich enough to absorb the per-phase "what shipped" content — which v1.0's was. For releases where the CHANGELOG stays terse, write the per-phase retros.
7. **Refusing the old shape before writing the new one is the right ordering for breaking releases.** ROS-81 made `roster init` hard-reject a `projects/` directory *before* substituting `config/project.yaml`. Users caught mid-upgrade get a single clear error instead of a half-applied workspace. Generalize: any breaking-shape release should have the refuse-the-old-shape guard *before* the write-the-new-shape code path.

## What to do differently next time

1. **Budget 30 minutes per phase close for a short retro, even when the release retro will roll them up.** Phases v1-2 / v1-3 / v1-4 didn't get per-phase retros; this file had to mine the CHANGELOG, `git log`, and Linear for content that would have been faster to write at phase close. A 30-line retro (tasks-shipped table + 2-3 decisions + 2-3 learnings) per phase compounds — and makes the eventual release retro a synthesis rather than an archaeology dig.
2. **Sequence doctor / audit-check tickets explicitly after the shape tickets they audit.** ROS-86 / 87 / 88 vs ROS-99 / 101 was a real ordering problem caught late. The dependency in `phases.yaml` should be explicit (`blocked_by: <shape-ticket>`), not implicit.
3. **Promote "deletion ticket owns its test cleanup" to a `/decompose` template invariant.** Encode it in `.forge/CLAUDE.md` so future decomposes don't reproduce ROS-76's gap. Cheaper than relearning it every release.
4. **Run `npm pack --dry-run` once per phase close, not just at release.** Phase v1-1 deferred tarball measurement to v1-4; in practice the tarball only moves on a few specific tickets (those touching `templates/scaffold/` or `skills/`). One `pack --dry-run` per phase catches regressions early at near-zero cost.
5. **Question whether 4 phases was the right granularity for this refactor.** Phases v1-2, v1-3, and v1-4 all closed on 2026-05-22 within hours of each other; the phase boundary between them was barely felt in execution. A future refactor of this size could plausibly run as two phases (shape + surface, then docs + release) instead of four — without losing planning clarity, because the contract was already settled in `conventions.md`.
