# Retro — v1.8.0: brain file system (ROS-156)

**Shipped:** 2026-07-08 (epic ROS-156; PRs [#308](https://github.com/firatcand/roster/pull/308) · [#309](https://github.com/firatcand/roster/pull/309) · [#310](https://github.com/firatcand/roster/pull/310) · [#311](https://github.com/firatcand/roster/pull/311))
**Scope:** S3-backed file store for the brain — `brain fs put|get|ls|rm`, an append-only `brain.files` ledger, a `FileStore` port over `@aws-sdk/client-s3`, and an `s3-file-drift` doctor check.

## What happened

The question was architectural: keep the brain Neon-only, or follow the athena
agent's **HydraDB + S3** pattern? We chose **Neon + S3** — Neon stays the
librarian and source of truth, S3 becomes the brain's file system — because
HydraDB has no SQL surface and can't host the brain's structured half (it would
mean Neon *plus* HydraDB *plus* S3, three vendors per user). ADR-0003 records
the decision and the rejected alternatives.

The epic was decomposed into four tickets and built in a Codex-gated loop:
plan → Codex second opinion → implement → Codex reviews the diff and scores →
fix everything real → repeat until pass. ROS-157 took one round, ROS-158 two,
ROS-160 two — and **ROS-159 took four**, each round peeling back a deeper
concurrency layer.

## What went well

- **The review gate found the bugs the tests didn't.** ROS-159's four Codex
  rounds walked a staircase: a put/rm race → a non-indexable-overwrite view
  gap → a get/rm false-drift → an rm delete-before-commit crash window. None of
  these showed up in the first (green) test suite; each became a regression
  test. The put/rm serialization test is proven to fail 5/5 with its lock
  removed — the review turned latent races into pinned invariants.
- **Verified against real infra, not just CI.** Installed MinIO locally and
  drove the actual built CLI end to end (put → query → get → rm → resurrect →
  drift) against real S3 + Postgres. That caught what a fake store never would
  and confirmed MinIO honors conditional PUT.
- **The view layer did the heavy lifting.** Making `current_documents`
  ledger-aware (`file_head.mount_id = latest.mount_id`) meant search, reindex,
  and gc became tombstone-aware with **zero** application changes — one SQL
  predicate instead of a join threaded through every reader.
- **A fake and a real backend held to one contract.** `MemoryFileStore` and
  `S3FileStore` run the same contract test; the fake couldn't drift from S3.

## What was hard

- **Concurrency across two stores.** The ledger (transactional Postgres) and
  the bytes (non-transactional S3) can't be committed atomically. The answer
  was a per-**address** advisory lock (not per-URI — an rm must lock before it
  reads the head, whose URI it doesn't yet know) held across the S3 op, plus a
  two-phase rm: durable tombstone first, then a re-checked delete. A crash can
  orphan an object (doctor-detectable) but never strand the ledger head over
  deleted bytes.
- **Durable-before-destructive is subtle.** The first rm deleted S3 inside the
  same transaction as the tombstone — a commit failure would have destroyed
  bytes the ledger still pointed at. Splitting it into two locked transactions
  fixed it without reaching for a pool-hazardous session lock.
- **Doctor's drift scan had to think in objects, not addresses.** A
  bucket/prefix change tombstones the old URI but re-puts the address, so an
  address-keyed orphan scan missed the abandoned object. Keying on `source_path`
  and filtering to the current bucket fixed both the miss and a latent
  false-positive "missing".

## What we'd do differently

- **Stacked PRs on an unmerged base are awkward.** ROS-159/160 had to carry
  their dependencies as merge commits and note "merge #308→#311 first." When
  siblings are this tightly coupled, either land the foundation before opening
  the dependent PR, or accept a combined diff — don't pretend the diff is clean.
- **The non-indexable-overwrite gap lived in the ROS-157 view but surfaced
  while building ROS-159.** A view invariant is only as good as the writes that
  exercise it; the ledger tests should have set `mount_id` from the start
  (matching real `putFile`) rather than leaving it null.

## Follow-ups

- **Media transcription** — audio/video/PDF are stored today but not
  searchable. Making them queryable (a transcription/parse step) is the natural
  next brain-file ticket.
- **Multipart upload** for very large media (deferred; single `PutObject` today).
